# Apache Airflow Learning Project

A structured, end-to-end learning project for **Apache Airflow 3.0**. The
repository progresses from "what is a DAG?" to "production-shaped pipelines"
through ten focused example DAGs, a small but real domain model, and a unit
test suite that locks the contracts in place.

```
airflow/
├── dags/                       # The DAGs themselves (this is the meat)
│   ├── 01_basics/              #   DAG anatomy, XCom, the smallest runnable DAG
│   ├── 02_etl/                 #   CSV → clean → SQLite, quality gates
│   ├── 03_orchestration/       #   Branching, dynamic generation, cross-DAG via Datasets
│   ├── 04_resilience/          #   Retries, callbacks, sensors, deferrable operators
│   └── 05_advanced/            #   TaskFlow, custom operators, hooks, an ML pipeline
├── include/                    # Shared, importable Python — the *heart*
│   ├── domain/                 #   Typed records (the schema)
│   ├── io/                     #   CSV and SQLite adapters
│   ├── transforms/             #   Pure functions over records
│   └── data/                   #   Sample CSVs
├── plugins/                    # Custom operators live here
│   └── operators/
├── tests/                      # Unit tests
│   ├── unit/                   #   Test the include/ package
│   └── dags/                   #   Test DAG structure (no scheduler needed)
├── docs/                       # Design notes (also tracked in Honcho)
├── .venv/                      # Local virtualenv (gitignored)
├── airflow.cfg                 # Generated by `airflow db migrate` (gitignored)
├── pyproject.toml              # Project metadata + tooling config
├── requirements.txt            # Airflow pin + a couple test extras
└── main.py                     # CLI helper: `python main.py demo` runs tasks
```

## TL;DR

```bash
# 1. Use a venv with Airflow installed (see "Setup" below).
export AIRFLOW_HOME=$(pwd)
python3 -m airflow db migrate

# 2. List the DAGs in this repo.
python3 -m airflow dags list --local

# 3. Run a single task on the command line — perfect for stepping with a debugger.
python3 -m airflow tasks test csv_to_warehouse extract.extract_orders 2024-06-01

# 4. The pytest suite covers the include/ package in milliseconds.
python3 -m pytest tests/unit/

# 5. The DAG integrity suite covers DAG loading + structure (no scheduler needed).
python3 -m pytest tests/dags/
```

## Setup

Airflow 3.0 recommends installing with the official constraints file so every
transitive dependency is pinned to a known-compatible version.

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip

# Choose a Python + Airflow combo from
# https://airflow.apache.org/docs/apache-airflow/stable/installation/installing-from-pypi.html
AIRFLOW_VERSION=3.0.2
PYTHON_VERSION="$(python3 -c 'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")')"
CONSTRAINT_URL="https://raw.githubusercontent.com/apache/airflow/constraints-${AIRFLOW_VERSION}/constraints-${PYTHON_VERSION}.txt"

pip install "apache-airflow==${AIRFLOW_VERSION}" --constraint "${CONSTRAINT_URL}"
pip install -r requirements.txt
```

Then point Airflow at this folder as its home and migrate the metadata DB:

```bash
export AIRFLOW_HOME=$(pwd)
python3 -m airflow db migrate
```

## Running DAGs

```bash
# Show every DAG this repo registers.
python3 -m airflow dags list --local

# Resolve a DAG id by listing its tasks (use this to discover task ids).
python3 -m airflow tasks list csv_to_warehouse

# Run a single task — bypasses the scheduler; perfect for `pdb`.
python3 -m airflow tasks test csv_to_warehouse extract.extract_orders 2024-06-01

# Trigger a full DAG run once.
python3 -m airflow dags trigger csv_to_warehouse

# Start the webserver + scheduler in one go (dev only).
python3 -m airflow standalone
```

The webserver defaults to <http://localhost:8080>. On a fresh install, Airflow
asks you to create the first admin user; follow the prompt.

## Running Tests

```bash
# Unit tests: ms-fast; covers include/ package only.
python3 -m pytest tests/unit/ -v

# DAG integrity tests: parse every DAG file in dags/, fail on missing
# owners, dangling tasks, cycles, and undefined references.
python3 -m pytest tests/dags/
```

## Honcho and design notes

Architectural decisions and the design rationale live in `docs/`. They are
also be tracked in a Honcho memory workspace called
`airflow-learning-project-redesign`, so the conversation context survives
across runs. Use the Honcho CLI skill to inspect the recorded conclusions:

```
/honcho-cli
```

The `docs/design_decisions.md` file mirrors the Honcho peer card for the
architect peer. If you change an architectural decision, update both.

## Debugging in PyCharm

The PyCharm run configurations from the original project still work; they
point at `dags/01_basics/airflow_demo.py` and an `airflow standalone`
configuration. Add new configurations for any DAG you'd like to step through.

```bash
# Equivalent of the PyCharm "Test Demo DAG" but on the command line:
python3 -m airflow tasks test airflow_features_demo processing_group.extract_data 2024-01-01
```

## The `main.py` entry point

`main.py` is a thin CLI that exposes the most common Airflow commands without
needing to remember the long `airflow` invocations:

```bash
python3 main.py demo              # run an example task (debugger-friendly)
python3 main.py list              # list registered DAGs
python3 main.py test <dag> <task> # equivalent to `airflow tasks test`
```

## Reading order

If you're new to Airflow, read in this order:

1. [`docs/taocp_applications.md`](docs/taocp_applications.md) — what we mean
   by "rigor" in this repo.
2. [`docs/design_decisions.md`](docs/design_decisions.md) — the contracts
   the `include/` package enforces.
3. [`dags/01_basics/`](dags/01_basics/) — see Airflow features in the wild.
4. [`dags/02_etl/`](dags/02_etl/) — see those features composed into a real pipeline.
5. [`dags/03_orchestration/`](dags/03_orchestration/) and beyond — building-block DAGs
   you can lift and adapt into your own workflows.
